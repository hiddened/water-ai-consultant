# AI 水务项目智能顾问平台演示脚本

## 演示目标

本次演示展示 `water-ai-consultant` 如何把水务项目资料转成可检索、可引用、可评测的知识体系，并完成从文档上传解析、智能问答、需求可行性分析、检索调试、评测回归到反馈转知识库的完整闭环。

重点讲清楚三件事：

1. 它不是普通聊天机器人，而是面向水务项目团队的内部智能顾问。
2. 回答不是纯靠模型生成，而是基于文档 chunk 和结构化知识检索。
3. 系统有引用溯源、评测和反馈沉淀能力，可以持续改进。

## 演示前准备

### 启动服务

```bash
cp .env.example .env
docker compose up -d --build
```

### 打开地址

- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:8080`
- Swagger：`http://127.0.0.1:8080/swagger-ui/index.html`
- MinIO 控制台：`http://127.0.0.1:9001`，仅在需要说明对象存储时展示

### 模型配置

- 演示生成式问答前配置 `LLM_PROVIDER=deepseek` 和 `DEEPSEEK_API_KEY`。
- 没有 API Key 时仍可演示知识管理、文档解析、检索调试和资料不足拒答，但不能演示有依据问题的模型生成结果。

### 可用示例问题

- 智能问答：`外部告警推送接口怎么用？`
- 需求分析：`客户想接入二供泵房告警，并在告警后自动生成维修工单，这个能不能做？`
- 拒答测试：`当前项目是否支持无人机巡检？`

## 演示路线

### 1. Dashboard

演示动作：

打开首页 Dashboard，展示项目数、文档数、文档切片、最近任务、评测用例、异常任务等概览。

演示话术：

“这是系统首页，定位是 AI 水务项目智能顾问的工作台。这里不是放营销信息，而是直接展示项目知识库和 AI 闭环的运行状态，比如当前有多少项目、文档、切片，最近任务是否成功，评测用例和异常任务情况。面试时我会从这里说明：这个系统关注的是内部项目交付效率和问答质量，而不是普通客服。”

可能追问：

问：Dashboard 的价值是什么？

答：它让用户一进来就知道知识库和任务是否健康，比如文档有没有解析、任务有没有失败、评测有没有覆盖。

问：这些指标从哪里来？

答：前端调用项目、文档、任务和评测相关接口汇总展示，底层数据来自 PostgreSQL。

### 2. 项目管理

演示动作：

进入“项目管理”，展示示例项目“智慧水务告警智能处置示范项目”，说明后续知识和问答都按项目隔离。

演示话术：

“项目是系统的核心隔离维度。水务行业不同客户项目的能力、接口和页面说明可能不一样，所以所有知识表基本都带 `project_id`。在这里可以维护项目基本信息，后续文档、页面说明、能力清单、接口说明、历史需求、问答记录和评测都会挂在某个项目下面。”

可能追问：

问：现在做了项目权限吗？

答：当前做了项目维度的数据隔离，但还没有完整用户登录和项目授权，这是后续生产化要补的。

### 3. 文档知识库

演示动作：

进入“文档知识库”，展示文档列表字段：解析状态、索引状态、切片数、最后解析时间、最后索引时间。上传 `sample-docs/` 中的示例文档，触发解析。点击“查看切片”。

演示话术：

“这里是文档知识库。用户上传项目文档后，后端会保存原文件，写入 `ai_document`，然后创建异步解析任务。解析成功后，Apache Tika 提取文本，`DocumentChunker` 把文本切片写入 `ai_document_chunk`。切片不是只保存内容，还保存文档标题、chunk 序号、页码或章节等来源信息，这样问答时才能引用到具体片段。”

可能追问：

问：为什么不直接把整篇文档给模型？

答：整篇文档太长且噪声大，切片后可以只召回相关内容，降低 Token 成本，也方便引用定位。

问：文档解析失败怎么办？

答：会更新 `parse_status=failed` 和 `parse_error`，同时任务中心写失败日志，不会让上传接口长期阻塞。

### 4. 任务中心

演示动作：

进入“任务中心”，筛选或打开刚才的文档解析任务，查看任务状态、进度、消息和日志。

演示话术：

“文档解析和索引都可能比较耗时，所以我没有让上传接口同步等待完整解析，而是设计了 `ai_task` 和 `ai_task_log`。上传接口返回 `task_id`，后台异步执行解析。前端任务中心可以轮询任务状态，查看进度和日志；失败任务可以重试，pending 任务可以取消。”

可能追问：

问：为什么第一版不用 MQ？

答：MVP 阶段用 Spring 异步和任务表足够简单可靠；后续大规模文档处理可以接 RabbitMQ 或 Kafka 做分布式任务。

问：任务状态有哪些？

答：主要有 pending、running、success、failed、cancelled。

### 5. 智能问答

演示动作：

进入“智能问答”，选择项目，输入问题：

```text
外部告警推送接口怎么用？
```

点击提问，展示回答、引用、相关页面、能力和接口。

演示话术：

“这是核心问答入口。用户提问后，后端不是直接把问题丢给模型，而是先根据项目和 mode 检索知识库。检索证据会转换成 Spring AI Document，由 PromptTemplate 组装上下文和引用，再通过 ChatClient 调用 DeepSeek。”

补充讲解：

- 检索文档切片：优先从 `ai_document_chunk` 召回相关片段。
- 组装 Prompt：`PromptTemplateService` 使用 Spring AI `PromptTemplate` 按 mode 渲染。
- 调用 DeepSeek：`ModelRuntimeService` 通过 Spring AI `ChatClient` 使用默认模型配置或环境变量。
- 返回引用来源：后端返回 `references` 并保存到 `ai_answer_reference`。

可能追问：

问：如何避免模型胡编？

答：没有检索依据时直接返回固定拒答，不调用模型猜；有依据时 Prompt 要求基于 references 回答。

问：为什么删除 Mock LLM？

答：Mock 输出容易让演示结果被误认为真实模型效果；现在生产链路只允许真实模型，无 Key 时保留非生成式功能并明确报错。

### 6. 引用详情

演示动作：

在问答结果中点击引用来源，打开原文详情弹窗，展示来源类型、标题、内容预览和完整内容。

演示话术：

“这一步是引用溯源。AI 应用在项目场景里不能只给结论，还要说明依据来自哪里。这里的引用可能来自文档 chunk、页面说明、能力清单、接口说明、数据表说明或历史需求案例。后端把检索证据统一转换成 references 返回，并落库到 `ai_answer_reference`，所以后续对话记录、反馈复核和评测都能追溯。”

可能追问：

问：引用存在数据库里有什么好处？

答：可以复盘每次回答的依据，也能在反馈和评测里判断回答是否引用了正确来源。

### 7. 需求可行性分析

演示动作：

进入“需求可行性分析”，输入：

```text
客户想接入二供泵房告警，并在告警后自动生成维修工单，这个能不能做？
```

点击分析，展示需求理解、可行性等级、已有能力、缺失能力、相关接口、风险点、推荐方案和引用。

演示话术：

“需求可行性分析是这个项目和普通文档问答最大的区别之一。客户问能不能做，不能只靠模型判断。后端会优先检索能力清单和历史需求案例，再补充页面、接口、数据库表和文档 chunk。模型输出结构化 JSON 后，后端还会做规则归一化：如果没有能力清单依据，就不能直接判断 A/B；如果没有任何依据，就只能返回 D：资料不足。”

讲解输出：

- 需求理解：模型把客户自然语言整理成业务目标。
- 可行性等级：A 已支持、B 配置支持、C 二开、D 资料不足、E 不建议。
- 已有能力：来自 `ai_capability`。
- 缺失能力：模型根据证据和需求列出。
- 相关接口：来自 `ai_api`。
- 风险点：来自历史案例或模型基于引用整理。
- 推荐方案：基于检索证据生成，不允许无依据承诺。

可能追问：

问：为什么只命中文档不能判 A/B？

答：文档描述不等于系统能力承诺，A/B 必须尽量依赖能力清单和历史案例。

问：资料不足怎么办？

答：返回 D，并建议补充项目文档、能力清单或历史需求案例。

### 8. 检索调试

演示动作：

进入“检索调试”，输入同一个问题或需求，设置 top_k，点击检索。展示命中的文档 chunk、能力、接口、页面、数据表、历史需求案例以及分数。

演示话术：

“如果问答不准，不能直接怪模型。这个页面是为了把 RAG 链路拆开看。它只做检索，不调用大模型。我们可以看到同一个问题到底命中了哪些 chunk、能力和接口，分数是多少，当前策略是 vector、keyword、hybrid 还是 fallback_keyword。如果这里没命中正确资料，问题可能在解析、切片或检索；如果命中正确但回答不好，再看 Prompt 和模型。”

可能追问：

问：检索调试能定位哪些问题？

答：能定位文档没解析、chunk 不合理、关键词没命中、向量检索失败、Prompt 组织问题和模型输出问题。

### 9. 问答评测

演示动作：

进入“问答评测”，展示评测用例列表。选择一条运行，或点击批量运行。

演示话术：

“问答系统不能只靠主观感觉好不好，所以我做了评测用例。每个用例可以维护问题、期望答案、期望来源、期望关键词、期望可行性等级和是否期望拒答。运行时系统会真实调用 `/api/chat` 或 `/api/requirements/check`，保存实际回答和引用，再按规则自动评分。”

可能追问：

问：评分怎么做？

答：主要按关键词命中、引用来源命中、拒答是否符合预期、可行性等级是否匹配来算分。

问：为什么还需要人工复核？

答：规则评分可解释但不覆盖所有语义质量，所以保留人工 passed、score、remark。

### 10. 评测运行看板

演示动作：

进入“评测运行看板”，打开一次批量运行，展示平均分、通过率、失败用例、分项得分和失败重跑。

演示话术：

“这里是回归测试看板。每次调整检索、Prompt 或模型后，都可以批量跑一遍评测，看通过率、平均分和失败问题。失败用例可以单独重跑。这样项目不是只实现功能，还能持续观察问答质量是否退化。”

可能追问：

问：这套评测和单元测试有什么区别？

答：单元测试验证代码逻辑，问答评测验证 AI 输出是否符合业务预期和引用要求。

### 11. 反馈管理

演示动作：

从问答结果提交一条反馈，进入“反馈管理”，展示反馈列表、筛选、详情、复核、转知识库。

演示话术：

“当用户发现回答错误、不完整或引用不准时，可以提交反馈。反馈进入 `ai_feedback` 后默认是 pending，管理员在这里复核。通过后可以转成知识库草稿，比如页面说明、能力清单、接口说明、数据表说明、历史需求案例或 FAQ。这样就形成了‘问答 -> 反馈 -> 复核 -> 知识沉淀’的闭环。”

可能追问：

问：怎么避免错误反馈污染知识库？

答：反馈不会直接启用，必须管理员复核；转换后的知识也保留 `source_feedback_id`，方便追溯。

### 12. 模型配置和 Prompt 模板

演示动作：

进入“模型配置”，展示 DeepSeek / OpenAI 兼容配置、测试连接、设默认。进入“Prompt 模板”，展示模板列表、编辑和测试渲染。

演示话术：

“模型和 Prompt 没有写死在代码里。模型配置支持 provider、模型类型、base_url、API Key、模型名、维度等，并且可以测试连接和设置默认。Prompt 模板按 template_type 和 mode 管理，比如文档问答、页面帮助、业务问答和需求分析可以用不同模板。模板测试可以预览变量渲染结果，方便定位 Prompt 问题。”

可能追问：

问：为什么 Prompt 要后台管理？

答：AI 应用上线后经常要调 Prompt，如果写死在代码里，每次调整都要发版；入库后可以更快调试。

问：API Key 安全怎么处理？

答：当前列表展示做了脱敏；生产上还需要进一步接密钥管理系统或环境变量注入。

## 演示收尾话术

“整体来说，这个项目不是简单调大模型，而是围绕水务项目资料做了完整 AI 应用闭环：前面有资料管理、文档解析和结构化知识；中间通过 Spring AI 串联 Document、PromptTemplate、ChatClient、结构化输出和 VectorStore；后面有引用、检索调试、评测回归、反馈复核和知识沉淀。”
