# 09 测试评测规则说明

## 测试类型

当前项目包含以下验证方式：

- 后端单元/集成测试：Maven `test`。
- 前端类型检查和构建：`npm run build`。
- API 联调：Swagger、curl 或前端页面。
- 问答评测：评测用例、自动评分、批量运行和回归看板。
- 检索调试：验证问题命中的知识来源。

## 后端测试

后端测试目录：

```text
backend/src/test/java
```

运行命令：

```bash
cd backend
mvn test
```

当前测试类为 Spring Boot 上下文基础测试，后续建议补充：

- Controller 参数校验测试。
- 文档解析状态流转测试。
- 检索排序和模式优先级测试。
- 需求等级归一化测试。
- 反馈转知识库测试。
- 评测自动评分规则测试。

## 前端测试

当前前端没有专门测试框架，主要通过类型检查和构建验证：

```bash
cd frontend
npm install
npm run build
```

`npm run build` 会执行：

- `vue-tsc -b`
- `vite build`

## 健康检查

本地或 Docker 启动后执行：

```bash
curl http://127.0.0.1:8080/api/health
```

成功响应说明后端进程可用。Docker Compose 也使用该接口作为后端健康检查。

## 问答评测对象

评测用例表：`ai_eval_case`。

核心字段：

- `project_id`：所属项目。
- `question`：测试问题。
- `expected_mode`：期望模式。
- `expected_answer`：期望答案说明。
- `expected_keywords`：期望命中关键词。
- `expected_source_titles`：期望来源标题。
- `expected_source_types`：期望来源类型。
- `expected_feasibility_level`：需求分析期望等级。
- `expected_refusal`：是否期望拒答。
- `tags`：标签。
- `difficulty`：难度。
- `enabled`：是否启用。

## 评测运行规则

支持接口：

- 单条运行：`POST /api/eval-cases/{id}/run`
- 批量运行：`POST /api/eval-cases/run-batch`
- 失败重跑：`POST /api/eval-runs/{id}/rerun-failed`

运行结果写入：

- `ai_eval_run`：批次统计。
- `ai_eval_result`：单条结果。

批次统计字段：

- `total_cases`
- `success_cases`
- `failed_cases`
- `average_score`
- `pass_rate`
- `model_provider`
- `model_name`
- `prompt_template_name`
- `search_strategy`

## 自动评分规则

评分由 `EvalScoringService` 完成，不调用模型打分。

非需求分析模式权重：

| 维度 | 权重 |
| --- | --- |
| 关键词命中 | 40% |
| 来源命中 | 40% |
| 拒答符合度 | 20% |

需求分析模式权重：

| 维度 | 权重 |
| --- | --- |
| 关键词命中 | 30% |
| 来源命中 | 30% |
| 拒答符合度 | 20% |
| 可行性等级 | 20% |

## 自动通过规则

默认自动通过线为 70 分，但还需满足：

- 如果 `expected_refusal=true`，回答必须包含固定资料不足拒答语。
- 如果配置了期望来源标题，但实际来源得分为 0，则强制不通过。
- 自动评分结果可通过人工复核修正。

## 来源评分规则

来源评分同时支持：

- 来源标题模糊匹配。
- 来源类型精确匹配。

没有配置期望来源时：

- 非拒答用例如果有引用，给基础来源分。
- 拒答用例不应返回引用；如果返回引用，来源得分为 0。

## 拒答评分规则

固定拒答文本：

```text
当前资料不足，无法确认
```

规则：

- 期望拒答时，实际回答包含该文本得 100，否则 0。
- 不期望拒答时，实际回答包含该文本得 0，否则 100。

## 人工复核规则

评测结果支持人工复核：

- `manual_passed`
- `score`
- `remark`

人工复核用于处理自动评分无法覆盖的语义正确性、表达质量和业务可接受性。

## 回归测试建议

每次修改以下内容后建议运行评测：

- 检索策略。
- Prompt 模板。
- 模型配置。
- 文档切片规则。
- 需求等级归一化规则。
- 反馈转知识库数据。
- 示例 seed 数据。

推荐流程：

1. 启动 Docker Compose。
2. 确认健康检查通过。
3. 上传或确认示例知识存在。
4. 执行批量评测。
5. 查看评测运行看板。
6. 对失败项查看缺失关键词、缺失来源和实际引用。
7. 修改知识或规则后重跑失败项。
