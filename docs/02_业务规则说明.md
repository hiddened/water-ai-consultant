# 02 业务规则说明

## 业务对象

项目围绕一个水务项目的内部知识进行管理，核心业务对象包括：

- 项目：`ai_project`，用于区分不同客户或演示项目。
- 文档：`ai_document` 与 `ai_document_chunk`，用于保存原文件信息、解析状态、切片和索引状态。
- 页面说明：`ai_page`，用于描述系统页面入口、操作路径和页面业务规则。
- 能力清单：`ai_capability`，用于描述系统是否支持某类能力，以及是否需要配置。
- 接口说明：`ai_api`，用于描述接口路径、请求、响应、鉴权和状态。
- 数据表说明：`ai_db_table`，用于描述业务表字段、关系和关键含义。
- 历史需求案例：`ai_requirement_case`，用于沉淀过往需求、方案、可行性等级、工作量和风险。
- 对话与引用：`ai_chat_session`、`ai_chat_message`、`ai_answer_reference`。
- 反馈与 FAQ：`ai_feedback`、`ai_faq`。
- 评测：`ai_eval_case`、`ai_eval_result`、`ai_eval_run`。
- 任务：`ai_task`、`ai_task_log`。

## 项目管理规则

- `project_code` 必须唯一，用于识别项目。
- `project_name` 必填。
- 项目支持 `status` 字段，当前前端默认使用 `active`、`archived`。
- 项目支持 `enabled` 和 `deleted`：
  - `enabled=false` 表示停用。
  - `deleted=true` 表示软删除，列表查询默认排除。

## 知识管理规则

- 页面说明、能力清单、接口说明、数据表说明、历史需求案例均必须归属项目。
- 上述知识均支持 `module_name`，用于按业务模块筛选。
- 知识数据删除时采用软删除，不物理删除业务记录。
- `enabled=false` 的知识不会进入检索和问答依据。
- 反馈转知识库生成的草稿默认 `enabled=false`，需要人工完善后再启用。

## 文档业务规则

- 上传文档必须提供 `project_id` 和非空文件。
- 文件名会替换 Windows/Unix 非法路径字符，避免本地存储路径异常。
- 文档上传只保存原文件和文档记录，默认 `parse_status=uploaded`，不会立刻生成切片。
- 文档解析通过异步任务执行，解析成功后生成切片，失败后保留原文件用于重试。
- 重新解析文档会删除该文档旧切片并重新写入，避免 `document_id + chunk_no` 唯一约束冲突。
- 索引失败不会删除切片，系统会保留关键词检索能力。

## 智能问答规则

智能问答支持以下模式：

- `doc_qa`：文档问答，优先使用文档切片和文档元信息。
- `page_help`：页面操作问答，优先使用页面说明。
- `business_qa`：业务问答，综合文档切片、能力、页面、数据表、接口和历史案例。
- `requirement_check`：需求分析模式，普通问答接口也支持该 mode，但需求分析推荐使用 `/api/requirements/check`。

问答必须先执行知识检索。没有检索依据时，系统返回固定拒答语：

```text
当前资料不足，无法确认。建议补充相关项目文档、能力清单或历史需求案例。
```

系统对“无人机”“区块链”“药耗优化算法”等明确专有能力做额外保护：如果问题包含该词但检索依据不包含同词，则按资料不足处理，避免过度泛化回答。

## 需求可行性规则

需求分析接口根据检索依据输出 `feasibility_level`：

- A：现有功能已支持。
- B：配置即可支持。
- C：需要二次开发或规则适配。
- D：资料不足，无法判断。
- E：不建议实现。

等级归一化规则：

- 无任何依据时必须为 D。
- 没有能力清单依据时，不能判断为 A 或 B，最多为 C。
- 没有同时命中能力清单和历史需求案例时，不能判断为 A，系统会降级为 B。
- 结论缺少能力清单依据时，会追加“未命中能力清单，不能判断为现有功能完全支持”提示。

## 反馈沉淀规则

- 反馈类型限定为 `useful`、`not_useful`、`wrong`、`incomplete`、`no_source`、`other`。
- 反馈默认进入 `pending` 复核状态。
- 反馈可复核为 `accepted`、`rejected`、`converted`。
- 勾选转知识库时必须指定目标知识类型。
- 支持转入的目标知识类型包括：`document_faq`、`page`、`capability`、`api`、`db_table`、`requirement_case`。
- 转知识库后生成草稿，默认不启用，避免未经复核的内容影响问答。

## 评测业务规则

- 评测用例必须包含 `project_id` 和 `question`。
- 评测模式通过 `expected_mode` 指定，可覆盖问答、页面、接口、需求分析等场景。
- 自动评分基于关键词、来源、拒答和可行性等级，不依赖再次调用模型打分。
- 自动通过分数线为 70 分，但来源缺失、拒答不符合预期时会强制不通过。
