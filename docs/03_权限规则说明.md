# 03 权限规则说明

## 当前权限现状

当前项目未实现登录、鉴权、用户、角色、菜单权限、数据权限或接口权限拦截。后端 Controller 没有使用 Spring Security，也没有基于 Token、Session、API Key 的统一鉴权过滤器。

因此当前运行状态为：

- 前端所有页面默认可访问。
- `/api/**` 接口默认可直接调用。
- Swagger 默认可访问。
- 模型配置、Prompt 模板、知识库维护、反馈复核、评测运行等管理能力没有角色限制。

该状态适合本地演示、学习项目、内网原型验证，不适合直接暴露到公网或生产环境。

## 当前隐含访问边界

虽然没有用户权限，但系统仍有以下业务边界：

- 多数知识数据通过 `project_id` 过滤，调用方需要主动传入项目 ID。
- 列表接口默认排除 `deleted=true` 的软删除数据。
- 问答检索默认只使用 `enabled=true` 且未软删除的数据。
- 文档问答只使用当前项目下的知识，不跨项目主动检索。
- 文件下载需要知道文档 ID，当前没有用户归属校验。

## 当前高风险接口

以下接口在未鉴权状态下风险较高：

- 模型配置：`/api/model-configs/**`，可新增、删除、设置默认模型，涉及 API Key。
- Prompt 模板：`/api/prompt-templates/**`，可改变问答行为。
- 文档管理：`/api/documents/upload`、`/api/documents/{id}/download`、`/api/documents/{id}/parse`。
- 知识维护：`/api/projects`、`/api/pages`、`/api/capabilities`、`/api/apis`、`/api/db-tables`、`/api/requirement-cases` 的写接口。
- 反馈转知识库：`/api/feedback/{id}/convert`。
- 评测运行：`/api/eval-cases/{id}/run`、`/api/eval-cases/run-batch`、`/api/eval-runs/{id}/rerun-failed`。

## 建议权限模型

后续生产化建议引入以下角色：

| 角色 | 建议权限 |
| --- | --- |
| 访客/只读用户 | 查看项目知识、问答、需求分析、查看引用 |
| 知识维护人员 | 上传文档、解析文档、维护页面/能力/API/表/案例 |
| 评测人员 | 维护评测用例、运行评测、人工复核评测结果 |
| 管理员 | 管理模型配置、Prompt 模板、反馈转知识库、任务重试 |
| 运维人员 | 查看健康检查、任务日志、部署配置和运行状态 |

## 建议接口权限分级

- 公开健康检查：`GET /api/health`。
- 普通用户：`POST /api/chat`、`POST /api/requirements/check`、只读知识列表和引用查看。
- 知识维护：文档上传/解析/重建索引、结构化知识 CRUD。
- 评测维护：评测用例 CRUD、评测运行、结果复核。
- 系统管理：模型配置、Prompt 模板、反馈转知识库。

## 数据权限建议

生产环境需要在当前 `project_id` 基础上补充用户-项目授权关系：

- 用户只能查看被授权项目。
- 问答、需求分析、检索调试必须校验用户是否有该 `project_id` 权限。
- 文档下载、切片查看、反馈查看、对话记录查看必须校验项目归属。
- 管理员可跨项目管理，但操作应记录审计日志。

## API Key 和密钥规则

当前模型配置会保存 `api_key_value`，列表和详情只返回 `api_key_masked`。编辑模型配置时，API Key 留空会保留旧值，避免误清密钥。

生产环境建议：

- API Key 加密存储，不明文落库。
- 禁止前端读取完整 API Key。
- 模型配置操作仅管理员可用。
- 日志不得打印 API Key、Authorization Header 或完整上游请求体。

## 部署访问规则

当前 Docker Compose 默认暴露：

- 前端：`3000`
- 后端：`8080`
- PostgreSQL：`5432`
- MinIO API：`9000`
- MinIO 控制台：`9001`

生产环境建议仅暴露前端或网关入口，后端、数据库、MinIO API/控制台均放入内网访问。
